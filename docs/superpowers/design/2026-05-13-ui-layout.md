# UI Layout — Motorsport Companion PWA

> Research-backed proposal for the screens Phase 1 already scaffolds. Identifies what to keep, what to tighten, and what to add as small, focused follow-up tasks.

## Constraints (from the spec, locked)

- Personal use, single user, installable PWA on Android.
- Dark theme by default. Each series gets a brand color used sparingly.
- "Session list is the hero — large, scannable, F1 Quali — Sat 16:00 (in 2 days)."
- App-like, not website-like. No marketing hero, no sponsor strip, no carousels.
- All times stored UTC, rendered Europe/Athens.
- ICS-driven sessions; sidecar `significance.json` flags marquee/finale/weighted/note rounds.

## Reference findings

Three sources reviewed, two extreme, one absent — the gap between them is where this app lives.

### 1. f1calendar.com — community-built F1 calendar (closest analog)
- List-style rows, one row per **session** (FP1, FP2, FP3, Q, Sprint, Race), grouped chronologically by weekend.
- Row content is minimal: event name + date + time-in-user-TZ. No flags, no weather, no countdown widget.
- Single accent: a "NEXT" label on the very next event for prominence.
- Lesson: dense, session-level granularity works — the user can scan a whole weekend without expanding anything.
- Gap: no series color, no significance flags, no day grouping, no past-session dimming.

### 2. formula1.com/en/racing/2026 — official, marketing-grade
- Card-style **weekend** layout: round number, country flag, hero image of the circuit, sponsor logos, big uppercase title.
- Practice/Quali/Race breakdown is hidden behind a per-race detail page.
- "NEXT RACE" label is the only highlight.
- Lesson (negative): the visual weight does not survive in a personal-use PWA. Hero images on every weekend would dominate the viewport on a phone and make scanning slower, not faster.
- One borrow: the **NEXT** label idea — bigger and more deliberate than f1calendar's tiny tag.

### 3. Material Design 3 — list components (intended reference, JS-rendered, no content returned)
- WebFetch returned the title only; M3 docs are client-rendered.
- Fallback to general M3 knowledge: 2-line list items with leading accent + trailing meta, 56px min height, 16px horizontal padding, dividers OR no dividers (not both), section headers in label-medium uppercase.
- This matches the SessionCard we built almost exactly — keep it.

## Information architecture per page

### Home (`/`)
**Hero question: "What's next?"**

Order, top to bottom:
1. **Header strip** — `Motorsport` (title) + `about` link. Sticky? Not for v1 — keep simple.
2. **NEXT** — a single, slightly larger card for the very next session across all series. Series-colored left bar, larger title, time + countdown, location, significance badge if any. Distinct from the normal SessionCard. *Not in current build.*
3. **This weekend** — section header + SessionList (Fri–Sun within the next 7 days). *In current build, no day separators.*
4. **Next 7 days** — section header + SessionList grouped by day with small "Sat 16 May" subheaders. *In current build, flat list.*
5. **Series** — compact row of SeriesBadge links. *In current build, vertical ul.*

### Series detail (`/series/[slug]`)
**Hero question: "What's the rest of the season look like, and why does it matter?"**

Order:
1. **Back-to-home affordance** + series badge + season heading + StaleBanner. *In current build, fine.*
2. **Calendar** — sessions grouped by **race weekend**, each weekend a small block: weekend title (e.g., "Monaco GP · 22–24 May"), sessions indented under it. Marquee/finale/weighted weekends get an accent tag and an amber left border. *In current build, flat session list — biggest UX gap.*
3. **Overview**, **Drivers**, **Significance** sections — markdown-driven, hidden when empty. *In current build, behaves correctly.*

### About (`/about`)
- Sources list with stale/fresh status + last-fetched timestamp. *In current build, fine.*
- Add a small "version / last build" footer line. *Not in current build, optional.*

## Component decisions

### SessionCard (existing — keep, refine)
Current:
- 4px series-colored left bar
- title (zinc-100), time + location (zinc-400)
- relative-time (zinc-500) right-aligned
- significance badge (amber) inline + note below

Refinements:
- Add **past** state: opacity-60, no relative-time hint ("past" suffices).
- Add **live** state: pulsing dot next to title, "LIVE" tag in series color (only if `start <= now <= end`).
- Bump min-height to 56px (M3 list item) so taps don't miss.

### NextSessionCard (new)
- 2x normal vertical padding
- Title at `text-lg` instead of `text-base`
- Countdown read as "in 2 days · Sat 22 May, 16:00 Athens"
- Series badge in the corner
- Tap → series detail page

### WeekendBlock (new — for series page calendar)
- Weekend header: weekend name + date range + significance tag if all-weekend
- Sessions list below, no top border (the block IS the visual grouping)
- Subtle alternating background for past vs upcoming weekends

### DayHeader (new — for home page next-7-days)
- One-line: "Sat 16 May" in `text-xs uppercase tracking-wider text-zinc-500`
- Sticky? No for v1, just inline.

## Design tokens (Tailwind v4 defaults, no custom theme yet)

| Token | Class | Use |
|---|---|---|
| Background | `bg-zinc-950` | App background |
| Primary text | `text-zinc-100` | Titles |
| Secondary text | `text-zinc-400` | Subtitles, times |
| Tertiary text | `text-zinc-500` | Hints, section headers |
| Muted | `text-zinc-600` | Empty-state copy |
| Divider | `border-zinc-800` | Row separators |
| Significance | `text-amber-300` / `bg-amber-900/40` | Marquee tag |
| Series accent | inline `style={{ backgroundColor: meta.color }}` | Left bar, badge dot |
| Live | `text-red-400` | LIVE tag |

Typography scale (matches Linear's density philosophy — no custom font, system stack):
- `text-lg` (18px) — app title, NextSessionCard title
- `text-base` (16px) — SessionCard title
- `text-sm` (14px) — subtitle, hint
- `text-xs` (12px) — section header (uppercase + tracking-wider)
- `text-[10px]` — significance badge

## Spacing rhythm

- Page padding: `p-4` (16px) on all sides, `max-w-2xl mx-auto` for tablet/desktop
- Section gap: `mb-8` (32px)
- Inline horizontal gap: `gap-3` (12px)
- Row vertical padding: `py-3` (12px) — bumps to ~56px row when title + subtitle render
- Border between rows: `border-b border-zinc-800` (Linear-style hairline)

## What to keep vs change

**Keep (matches research + spec):**
- Dark-only theme
- Series-colored left bar on rows
- Section-headered vertical list
- Sources + stale/fresh About page
- ISR for data freshness

**Tighten (small):**
- SessionCard min-height 56px
- Past sessions dimmed
- Section headers consistent uppercase+tracking

**Add (small follow-ups, each ~30 lines of code):**
1. `NextSessionCard` + render it above "This weekend" on home
2. `DayHeader` + day-grouping in the next-7-days list
3. `WeekendBlock` + weekend-grouping in series-page calendar
4. Past-session dimming + LIVE state (~10 lines in SessionCard)
5. Significance overlay onto the WeekendBlock header (not just per-session) for "the whole weekend is the marquee"

**Defer (Phase 2+):**
- Filter chips for series (waits for multiple series in v2)
- Bottom nav (waits for `/calendar` page in v3)
- Pull-to-refresh / install prompt UX (Phase 4)
- Real icon set (Phase 4)
- Calendar grid visualization (probably never — list is the spec'd hero)

## Implementation: 5 small tasks for a UI follow-up plan

If the user wants to act on this:
1. SessionCard refinements (past dim, LIVE state, min-height) — 1 commit
2. New `NextSessionCard` + wire into `/` — 1 commit
3. New `DayHeader` + day-grouping helper in `lib/date.ts` + wire into `/` — 1 commit
4. New `WeekendBlock` + weekend-grouping helper (probably in `lib/series.ts` or `lib/group.ts`) + wire into `/series/[slug]` — 1 commit
5. Whole-weekend significance overlay on `WeekendBlock` — 1 commit

Each is independent and reversible; no big-bang redesign.

## Open questions for the user

1. **NEXT card placement** — above "This weekend" (always present) or only when there's nothing this weekend (so it doesn't duplicate)?
2. **Weekend grouping** — automatic by clustering sessions within 4 days, or require an explicit weekend-id field in `significance.json` (e.g. `{"2026-05-24": {"weekend": "Monaco GP", "tier": "marquee"}}`)?
3. **Series page calendar default view** — full season scroll (current) or collapsed to upcoming-only with a "show past" toggle?

## Sources

- [F1 Calendar 2026 — homepage](https://f1calendar.com/) — session-level list, "NEXT" label, no visual extras. The closest model for this app.
- [Formula1.com 2026 schedule](https://www.formula1.com/en/racing/2026) — card-style weekend layout with circuit hero images. Reviewed as the negative example (too marketing-heavy for a personal PWA).
- [Material Design 3 — Lists](https://m3.material.io/components/lists/overview) — intended reference for list-item density and structure; the page is client-rendered so content was not retrievable via WebFetch. Conclusions based on general M3 list-item knowledge (56px min, 2-line layout, leading/trailing slots).
- [Linear Web Dark — Mobbin screen](https://mobbin.com/explore/screens/4e98c641-4c8f-44c9-aafa-05c229578827) — referenced for dark-mode list density philosophy (hairline dividers, tight type scale, restrained accent color).
- [Linear redesign notes — "How we redesigned the Linear UI"](https://linear.app/now/how-we-redesigned-the-linear-ui) — referenced for the philosophy of "increase the hierarchy and density of navigation elements." Drove the choice to keep sections vertical and minimal rather than introducing tabs/bottom-nav.
