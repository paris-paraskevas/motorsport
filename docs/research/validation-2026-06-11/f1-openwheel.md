Triple-confirmed: matches the Wikipedia 2025-26 season article (prod's own upstream — circular on its own) AND independent press standings (AutoHebdo via FIA/RacingNews365 search results, https://www.autohebdof1.com/news/car/formula-e/formula-e-2026-rankings-e-price-of-monaco.html) exactly. Driver rows show no team affiliation — by design (Wikipedia's FE drivers table has no team column; code comment says curate `content/series/formula-e/drivers.json` to surface teams — file still absent, cosmetic).
- **RESULTS: PASS (minor note).** Prod lists 10 rounds (São Paulo 6 Dec → Monaco R10 17 May) — exactly the 10 of 17 E-Prix run, doubleheader numbering correct. Most recent (Monaco R10) podium: Rowland (Nissan) / Drugovich (Andretti) / da Costa (Jaguar) — exact match vs Wikipedia. Minor: a couple of rounds (R6 Madrid, R9 Monaco in this fetch) render abbreviated classifications (~top-4) rather than full top-10 — consistent with the documented winners-only/motorsportweek fallback layering for rounds whose Wikipedia per-event articles are stubs; known degradation, not wrong data.
- **CHAMPIONS: DISCREPANCY — trust-killer (dead surface).** Prod renders the error state: "No champions data. We couldn't parse a champions table for Formula E." with only a View-on-Wikipedia link. Official 3 most recent that should display: 2024-25 Oliver Rowland (Nissan), 2023-24 Pascal Wehrlein (Porsche), 2022-23 Jake Dennis (Andretti). Root cause (code-level): formula-e has NO curated `champions.json` (unlike f1/f2/f3), so `components/tabs/ChampionsTab.tsx:352` falls back to live-scraping `List_of_Formula_E_champions` via `lib/wikipedia-champions` (`content/series/formula-e/meta.json:9`), which parses 0 rows — that page's champions table shape (season-format "2014-15" first column, cross-annotated tables) evidently defeats the generic parser. Fix options: curate `content/series/formula-e/champions.json` (fastest, matches the authoring model) or teach the parser season-range years.
- **Source solidity:** `lib/standings/formula-e.ts` + `lib/results/formula-e.ts` → en.wikipedia.org season-article scrape (fiaformulae.com is a JS shell; no public API), revalidate 3600s, fail-closed floors (≥12 drivers / ≥6 teams), colspan-unfolding for doubleheader columns; results fall back per-round to winners-only rows, then motorsportweek.com per-event tables (points DERIVED from position there — no pole/FL bonuses, documented ±1-3pt risk vs standings). Fragility: editor-dependent upstream — table-shape edits or team-name edit wars (DS Penske/Citroën alias map exists in code) are the live risk; champions parser already demonstrates the failure mode.

## indycar

- **STANDINGS: PASS.** Prod top-5: Palou (Ganassi) 342 (4 wins), Kirkwood (Andretti Global) 293 (1), Malukas (Penske) 274, Lundgaard (Arrow McLaren) 246 (1), O'Ward (Arrow McLaren) 239 — triple-confirmed exact vs Wikipedia 2026 IndyCar Series AND independent press (racetrackmasters/total-motorsport post-Gateway standings, surfaced via web search). Known scope limit: no teams/entrants table on prod (`lib/standings/indycar.ts` returns drivers only) while an official Entrant standings exists — absent surface, not wrong data; cosmetic.
- **RESULTS: DISCREPANCY — trust-killer.** Race count and latest race are right: 9 races listed = 9 run (St. Pete → Gateway 7 Jun); Gateway podium Newgarden (Penske) / Ericsson (Andretti) / Rasmussen (ECR) exact vs official (https://www.indycar.com/results/ntt-indycar-series/2026/bommarito-automotive-group-500/race; +0.6613s / +1.8523s). **But the 110th Indianapolis 500 block is garbled:** verbatim prod rows run P5 Armstrong, P9 Grosjean, P10 Sato, P11 Siegel, P13… — positions 1, 2, 3, 4, 6, 7, 8, 12 are missing from the visible top-10. The actual winner Felix Rosenqvist (closest finish in Indy history, 0.0233s over Malukas; P3 McLaughlin, P4 O'Ward, P5 Armstrong per indycar.com/ESPN) is absent, so the card reads as if Armstrong won the Indy 500. Root cause (high confidence, code-level): Wikipedia's matrix decorates Indy 500 cells with SUPERSCRIPT NUMERALS for qualification points (top-12 qualifiers); `parseCell` in `lib/results/indycar.ts` reads `cell.text()`, which flattens `<sup>12</sup>` into the digits — "1¹²" becomes "112" and `match(/^(\d{1,3})/)` (line ~255) yields phantom position 112 (or "2⁶" → 26, etc.). Phantom positions sort out of the top-10 window (or into wrong slots) AND feed `computePoints` (P>25 → 5 pts), so the affected 8 drivers' Indy points corrupt any season-trend totals too. The missing-position set (1-4, 6-8, 12) matches "top-12 qualifiers who finished in the visible window" exactly. Fix: strip `<sup>` (digit) nodes before text extraction in `parseCell` — the `<sup>L</sup>` led-laps marker is already handled separately.
- **CHAMPIONS: PASS.** Prod: 2025 Álex Palou (4th title), 2024 Álex Palou (3rd), 2023 Álex Palou (2nd), all Chip Ganassi Racing — matches the official record.
- **Source solidity:** standings → indycar.com `/Standings` SSR scrape (`data-driver-data` JSON attrs, Chromium UA, revalidate 3600s, fail-closed ≥10 floor); results → Wikipedia 2026_IndyCar_Series driver-standings matrix with points RECOMPUTED from a hardcoded 2026 scale + pole/led-laps bonuses (most-laps-led +2 not modelled — documented gap), hardcoded schedule map for names/dates (editorially weak June-30 fallback date), revalidate 3600s. Fragility: matrix-cell decoration parsing is the live risk — the Indy quali-superscript bug above is exactly this class; any new Wikipedia footnote convention can silently corrupt positions again.
