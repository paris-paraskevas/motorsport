2. **Clerk SDK (~288 KiB / ~224 unused)** — loads eagerly because `AppShell` (all-client) mounts Clerk hook consumers (`HeaderUtils`, `OnboardingWizard`) + the anon `SignInButton` on every route. → defer for anonymous visitors (push the boundary down / `next/dynamic` / `<ClerkLoaded>`). **M, ~224 KiB for the dominant anon cohort.**
3. **GTM/GA (~154 KiB)** — already `lazyOnload`. Fine.

- **CSS critical path** (~2 render-blocking bundles to ~2s) — re-measure after the above; likely shrinks once AdSense/Clerk drop.
- **Bundle analyzer** (`next experimental-analyze`) — attribute ~160 KiB of unlabeled 1st-party chunks. S, diagnostic.

## 3. Faster results loading

### Current flow + bottlenecks
- `/api/just-missed` (force-dynamic, CDN `s-maxage 300` / SWR 600) fans out in parallel over 6 series; each podium KV-cached 3h — but **no warm path** (only lazily populated by a cold visitor). Cold (edge + KV both cold) → dozens of upstream calls → ~14s TTFB.
- **WEC** is the slowest source: a sequential live-component POST chain (~5 POSTs/round, rounds sequential) ≈ 22 serial CMS round-trips on a cold miss.
- **MotoGP** has no parser-level KV cache (re-fetches the whole season every cold miss).
- The **health cron** runs the parsers every 6h for liveness but **discards** the output.
- Uncached Wikipedia scrapers: Formula E, WRC, NASCAR. `withSourceSnapshot` (durable last-good) is wired to **news only**, not results.

### Recommendation (prioritized)
1. **[S] Warm-results cron** — `app/(app)/api/cron/warm-results` (cron-auth'd) loops the home-supported series, calls `fetchLatestPodium(slug)` (reuses the existing write-through) to pre-populate `paddock:home:podium:*` every ~15–30 min (tighter on race weekends) + re-prime the edge. Kills the cold path structurally. **Highest ROI.**
2. **[S] MotoGP parser-level KV cache** (mirror f2/f3/wec) + **[S] reuse the health cron's compute** to fill the cache instead of discarding it.
3. **[M] One normalized "latest results" snapshot** (`paddock:home:latest-all`) the route reads in a single KV get → zero per-series work on the request path; store via `withSourceSnapshot` for durable last-good.
4. **[M] Extend `withSourceSnapshot`** (or last-good) to the slow scrapers (WEC, Formula E, WRC, NASCAR) so a cold-miss-during-outage degrades to last-good, not blank.

## Appendix — can a PWA have home-screen widgets (Android / iOS)?

**No (as OS home-screen widgets), as of 2026.** Neither Android nor iOS supports true home-screen widgets for PWAs. The `widgets` web-app-manifest member is a proposal implemented only on **Windows** (the Edge Widgets Board, via Adaptive Cards); the **Android** Chromium "Fugu" request is open but untriaged/unshipped; **iOS** is WebKit-gated and PWAs can't update content in the background, so widget-style live data isn't possible.

- Paddock's in-app "home widgets" (the dashboard blocks) work fine in the installed PWA — that's just web UI, unaffected.
- The practical home-screen lever today is **manifest `shortcuts`** (deep-link menu on long-press of the app icon; Android WebAPK + desktop).
- True OS widgets would need a **native / TWA wrapper** on Android (the parked W6 item) and aren't possible on iOS without a native app.

Sources: [Progressier — can a PWA create home screen widgets](https://intercom.help/progressier/en/articles/12814946-can-a-pwa-create-home-screen-widgets) · [Chromium Fugu request (Android)](https://issues.chromium.org/issues/40744630) · [web.dev — PWA enhancements/shortcuts](https://web.dev/learn/pwa/enhancements) · [MobiLoud — PWAs on iOS 2026](https://www.mobiloud.com/blog/progressive-web-apps-ios).
